Opening

Converting completion-handler code to structured concurrency is showing up in our crash logs and slow CI jobs — not just as compile edits but as changed cancellation and lifetime semantics that surface at runtime. Treat these migrations as operational work: add tests that assert cancellation, instrument async boundaries, and stage rollouts so you don’t turn a refactor into an availability incident.

This Week's Big Story

Structured Concurrency Patterns for Swift 6.3

Migrating callback-heavy flows to async/await can silently change who owns work, when children are cancelled, and whether network requests are torn down properly. Those differences are the reason you’ll see leaked tasks or UI races only after shipping. Plan tests, observability, and incremental rollouts as part of the rewrite.

Swift Snippet of the Week

// ✅ Preferred: bounded, aggregate results func fetchAll(_ urls: [URL]) async throws -> [Data] { try await withTaskGroup(of: Data?.self) { group in for url in urls { group.addTask { let (data, _) = try await URLSession.data(for: URLRequest(url: url)) return data } } var results: [Data] = [] for await maybeData in group { if let data = maybeData { results.append(data) } } return results } }

This encodes an engineering judgment: bind child work to the caller, keep results aggregated and bounded, and avoid detached tasks that outlive the intended scope.
