## iOS Dev Weekly — Issue #7

### Opening

SwiftUI makes UI composition easier, but rendering regressions still ship and surprise teams. The hard part isn't that SwiftUI can be slow—it's that slow frames, CPU spikes, and creeping memory use are only useful data when you can reliably reproduce and attribute them. This issue walks through a practical, repeatable way to profile SwiftUI rendering with Instruments so your findings are actionable.

---

### This Week's Big Story

**Profile SwiftUI Rendering with Instruments**

UI regressions often show as blurred frames, CPU spikes during scrolling, or a steady memory climb after a release. Effective investigation requires profiling release builds on a device, marking meaningful boundaries in your code, and correlating traces to user actions. Do that once and you stop guessing—your next rollouts can be surgical instead of rollbacks.

### Swift Snippet of the Week

```swift
import SwiftUI
import Observation
import OSLog

let profilerLogger = Logger(subsystem: "com.example.app", category: "rendering")
let signposter = OSSignposter(logger: profilerLogger)

@Observable class ListViewModel {
    var items: [String] = (1...200).map { "Item \($0)" }
}

struct ItemRow: View {
    let text: String
    // transient UI state lives here to avoid bubbling changes up the hierarchy
    @State private var highlighted = false

    var body: some View {
        Text(text)
            .padding()
            .background(highlighted ? .gray.opacity(0.2) : .clear)
            .onTapGesture {
                // mark a user-driven boundary for Instruments
                let id = signposter.makeSignpostID()
                signposter.beginInterval("RowTap", id: id, "row", text)

```

This pattern matters because targeted signposts and localized transient state make Instruments traces mappable to user actions and reduce noisy recomputation across your view tree.
